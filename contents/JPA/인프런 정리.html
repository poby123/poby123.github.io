Entity가 API 스펙에 영향을 줘서는 안된다.
  * 문제
      * 1. Entity의 변수 이름을 바꾸거나 하면 API 스펙이 바뀌게 된다.
      * 2. 다양한 API가 있는데, 어떤 API는 어떤 필드가 NotEmpy이어야 하지만, 다른 필드는 아닐 수도 있다.
      * 3. Entity가 화면 계층에 의존적이 되면, 양방향으로 의존 관계가 생기므로, 유지 보수가 어려워진다.
      * 4. 막을 수는 있지만 기본적으로 Entity의 모든 내용이 노출된다. 

  * 해결 방법
      * 이를 위해 DTO(Data Transfer Object)를 도입한다.
      * DTO에 Validation 어노테이션을 적어주면 조금 더 명확해지는 장점이 있다.
      => Entity와 프레젠테이션 계층을 위한 로직을 분리해야 한다.
      => Entity를 밖에 노출하지 말아라.
